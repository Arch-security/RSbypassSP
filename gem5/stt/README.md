# STT-Based PoC Demonstration

This directory contains the STT-based simulator and PoC code used for the Section 5 evaluation in our paper：

> Section 5: Exploiting taxonomy-overlooked μop expansion under STT (Limitation I & II)

In the paper, Section 5 evaluates the PoC demonstrations introduced in Section 4. The goal is to show that speculative RS contention remains inducible under the strongest security-oriented configurations of state-of-the-art selective-speculation defenses. This directory corresponds to the STT part of that evaluation.

Our STT simulator is based on the publicly released STT artifact:

<https://github.com/cwfletcher/stt>

The original STT artifact implements the defense proposed in the MICRO'19 paper:

> Speculative Taint Tracking (STT): A Comprehensive Protection for Speculatively Accessed Data

## Directory Overview

The relevant PoC code is located at:

```text
stt/RS_Contention_Taxonomy-overlooked
```

This PoC demonstrates taxonomy-overlooked μop expansion under STT.

Specifically, the PoC leverages the difference in the number of micro-operations generated by the `REP MOVSB` instruction under different `RCX` values. It first uses the predicated instruction `CMOVNE` to generate a secret-dependent `RCX` value. Then, by executing `REP MOVSB` with this secret-dependent `RCX`, it induces secret-dependent Reservation Station (RS) contention.

The induced RS contention delays the fetch of subsequent instructions, forming an I-cache side channel in gem5 x86. This demonstrates that STT can miss this RS-contention channel because it is outside the original transmit-instruction taxonomy.

## Docker Setup for STT

For artifact evaluation, we recommend using the provided Docker environment.

The included Docker image uses `ubuntu:20.04` with the following main dependencies:

- `python2` / `python2-dev`
- `gcc-9` / `g++-9`
- `protobuf-compiler`
- `libprotobuf-dev`
- `libpng-dev`
- `zlib1g-dev`
- `musl-tools`
- `SCons 2.5.1`

The current recommended x86 build target is:

```text
build/X86/gem5.opt
```

The current `build/X86` default protocol has been aligned to:

```text
MESI_Two_Level
```

## 1. Build the Docker Image

From the repository root, run:

```bash
docker build -t stt-ubuntu20-py2 .
```

If you have already run `./gem5/install.sh` or `./gem5/install.sh stt` from the
artifact root, this Docker image has already been built and this step can be
skipped.

## 2. Start a Docker Shell

From the repository root, run:

```bash
docker run --rm -it -v "$PWD":/workspace/stt stt-ubuntu20-py2
```

Inside the container, go to the STT repository root:

```bash
cd /workspace/stt
```

## 3. Build STT/gem5 for X86

Use the repository `Makefile` target:

```bash
make x86-64
```

This is equivalent to:

```bash
scons --config=force build/X86/gem5.opt
```

The `--config=force` flag refreshes SCons configuration checks and avoids stale
`build/.scons_config` results from a previous host or container environment.

If the custom x86 ISA files are modified, rebuild this target again.

## 4. STT Configuration

The STT simulator supports the following configuration options.

### `--threat_model [string]`

Different threat models:

- `UnsafeBaseline`: unmodified out-of-order processor without protection.
- `Spectre`: Spectre threat model, covering control-flow speculation.
- `Futuristic`: Futuristic threat model, covering all types of speculation, exceptions, and interrupts.

### `--needsTSO [bool]`

Configure the consistency model:

- `True`: use Total Store Ordering, or TSO.
- `False`: use Relaxed Consistency, or RC.

### `--STT [int]`

Configure STT:

- `0`: disable STT. In this case, the defense blocks all speculative transmitters.
- `1`: enable STT.

### `--implicit_channel [int]`

Configure implicit-channel protection:

- `0`: ignore implicit channels.
- `1`: enable protection against implicit channels.

### `--moreTransmitInsts [int]`

Configure the set of arithmetic micro-ops that are treated as transmitters:

- `1`: integer divide (`idiv`) and floating-point divide.
- `2`: integer divide (`idiv`) and all floating-point micro-ops.

## 5. Custom `REP MOVSB` Implementation

To better match real CPU behavior for `REP MOVSB`, we use a branch-less implementation that selects fixed micro-op paths for different `RCX` ranges.

This implementation is used by the PoC in:

```text
RS_Contention_Taxonomy-overlooked
```

## 6. Build the PoC Sample Statically

The sample under `RS_Contention_Taxonomy-overlooked` is expected to be built as a static x86 binary.

The provided Docker image already includes `musl-tools`.

Build the sample with `musl-gcc`:

```bash
cd /workspace/stt/RS_Contention_Taxonomy-overlooked
make clean
make CC=musl-gcc all
```

The generated binary used by the STT demonstration is:

```text
RS_Contention_Taxonomy-overlooked/v1REP_evaluation
```

## 7. Run the PoC Through the Repository Makefile

Go back to the repository root:

```bash
cd /workspace/stt
```

Run the PoC through the repository `Makefile` target:

```bash
make v1rep
```

This target uses:

- `build/X86/gem5.opt`
- `configs/example/se.py`
- `RS_Contention_Taxonomy-overlooked/v1REP_evaluation`

The Makefile target is expected to run the following PoC binary under gem5:

```text
RS_Contention_Taxonomy-overlooked/v1REP_evaluation
```

## 8. Expected Output

A successful run should execute the PoC binary under the STT-based gem5 simulator.

The output should look similar to the following:

```text
Global frequency set at 1000000000000 ticks per second
Info: simulation uses threatModel: Futuristic; needsTSO=0
applySTT = 1, implicit_channel = 1, ifPrintROB = 0, moreTransmitInsts = 2
**** REAL SIMULATION ****
0,56
1,197
0,46
1,165
0,11
1,166
0,12
0,11
Exiting @ tick 2607379000 because exiting with last active thread context
```

In this example, the PoC successfully leaks the character:

```text
'T'
```

whose binary representation is:

```text
01010100
```

The lines printed by the PoC correspond to the recovered bit values and their associated timing/probe measurements. The important qualitative result is that the secret-dependent `REP MOVSB` behavior induces distinguishable RS-contention effects even when STT is enabled under the Futuristic threat model.
